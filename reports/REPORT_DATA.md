## B. ДАННЫЕ (HMDB51)

- **Классы (10), точный список** (`src/config.py:CLASSES`):
  `climb, draw_sword, fencing, golf, pullup, ride_bike, run, shoot_bow, walk, wave`
  (две намеренно спорные пары: `run`/`walk`, `fencing`/`draw_sword`).
- **Источник:** HuggingFace dataset **`jili5044/hmdb51`** — один файл `hmdb51.zip` (2118 MB),
  внутри — клипы по папкам-классам. Запасной источник: оригинал serre-lab (RAR, `--source serre-lab`).
- **Лицензия:** HMDB51 — для исследовательских/учебных целей (Brown University, serre-lab);
  клипы из публичных источников и фильмов. В репозиторий датасет **не** включён.
- **Формат файлов:** `.avi` (видео), одна папка на класс.
- **Тип разметки:** video-level classification — **один ярлык-действие на ролик**.
- **Всего клипов (сырые, `download_hmdb --check`):** 1634.
  Использовано после капа и разбиения: **1154** (train 694 / val 114 / test 346).

- **Метод разбиения** (`src/data/splits.py`): официальных HMDB51-сплитов в зеркале нет →
  **свой стратифицированный split с фиксированным seed=42**. По классам: список `.avi`
  сортируется и перемешивается (seed), обрезается до `MAX_PER_CLASS=150`, затем
  `test_ratio=0.3` (test = первые 30%), из оставшегося train выделяется
  `val_ratio=0.15` (стратифицированно, seed). Официального val у HMDB51 нет.
- **Дисбаланс:** `walk` (548) и `run` (232) на порядок крупнее прочих (~100-116);
  кап `MAX_PER_CLASS=150` подрезает их до 150, мелкие классы сохраняются целиком.
- **Аугментации** (`src/data/dataset.py`):
  - *temporal sampling*: равномерное деление на `num_frames` сегментов, 1 кадр/сегмент,
    джиттер (train) / центр (eval); короткие клипы зацикливаются;
  - *resize*: меньшая сторона → `round(img_size*1.15)`, билинейно;
  - *crop*: случайный (train) / центральный (eval) до `img_size`, единый бокс на все кадры;
  - *horizontal flip*: случайный, p=0.5 (только train);
  - *normalize*: mean `(0.45, 0.45, 0.45)`, std `(0.225, 0.225, 0.225)`.

## H. АНАЛИЗ ОШИБОК

- **Главная путаница — `run`↔`walk`**: у VideoMAE это **20 из 43** ошибок
  (`walk→run` 12 + `run→walk` 8). Причина: оба класса — локомоция, отличаются в основном
  **темпом**, который при 8–16 кадрах легко спутать; в top-3 верный класс обычно второй.
- **`walk` и `run` — худшие по F1** (VideoMAE: walk 0.6437, run 0.7174); `wave` тоже
  частично уходит в `walk` (5×).
- **`fencing`/`draw_sword` модель различает уверенно** (VideoMAE F1 0.904 / 0.967) —
  «мечевая» пара визуально расходится сильнее, чем ходьба/бег. (У I3D, наоборот, заметна
  `fencing→run` 10 — у более слабой 3D-модели путаются и эти классы.)
- **TSN/TSM слабы** (acc ≈ 0.51–0.52): backbone предобучен на ImageNet (статичные кадры),
  а при замороженном backbone голова не выучивает движение; у 3D-CNN/VideoMAE backbone
  предобучен на Kinetics (видео-действия), потому «только-голова» уже силён.
- **Идеи по улучшению:** (1) **полное дообучение backbone (`--no-freeze`)** — особенно
  поднимет TSN/TSM; (2) больше кадров / выше FPS-сэмплинг для разведения `run`/`walk` по
  темпу; (3) взвешивание классов / oversampling сложных; (4) оптический поток как доп.
  модальность для близких по виду движений.

## K. ДЛЯ ВЫВОДОВ

- **Лучший по точности:** **VideoMAE** — accuracy **0.8757225433526011**,
  F1(macro) **0.8922175416325316**, stability **1.0** (самый тяжёлый: 86.2M params,

- **Лучший для практики на CPU:** **I3D** — accuracy **0.7716763005780347**,
  F1 **0.7951520241454185** при вдвое меньшем размере (104 MB, 27.2M) и наименьшей

- **Слабые:** TSN (0.5231) и TSM (0.5145) — ограничены замороженным ImageNet-backbone.
- **Влияние длины клипа:** для 3D-CNN точность растёт до плато ~16 кадров.
- **Применимость:** систему **реально применять** для распознавания действий на коротких
  роликах — топ-модель уверенно различает большинство из 10 классов (F1 ≈ 0.89); узкое
  место — близкие по динамике действия (`run`/`walk`).
